# Prompt — Field-indexed backend dependency list (gap/priority finder)

> **This is now generated by a script — hand-running this prompt is no longer necessary.**
> `python fedMigrationScripts/generatescripts/generate_client_story_dependency.py` (also runs inside
> `generate_all.py`) regenerates every file in `output/clientStoryDependency/` wholesale from
> `frontend-inventory.json` + `fe-08-frontend-stories.md` + each domain's `be-04/be-05` docs +
> the raw `ClientCallingGqlQueries/*.txt` sources — no Claude session required. This file stays as
> background/rationale documentation (the worked example and field-indexing philosophy below are
> still the best explanation of *why* the output looks the way it does) but is no longer the
> operational how-to.

> **Use when:** for each frontend query operation, you need to know what backend stories/entities
> resolve each field it depends on — indexed BY FIELD, not by query. One row per distinct
> field/entity, with a column listing every query operation it impacts. This surfaces external
> dependencies and missing stories in one place without repeating the same fact once per operation.
>
> **The deliverable is a gap/priority list, field-indexed.** Keep every file short: one table per FE
> story, one row per distinct field/entity, done. No narrative, no separate query table, no separate
> Flags table — this single table carries everything.
>
> **Worked example (target level of detail):** for `PRODUCT-FE-001`, instead of one row saying
> "`GET_SPARK_PRODUCT_SCAFFOLDING` → depends on 9 stories," the table has one row for
> `VMM_BusinessPartner` (Story: `NEW-PRODUCT-BE-H-??`, External: VMM, New: Yes, Impacts:
> `GET_SPARK_PRODUCT_SCAFFOLDING, GET_PRODUCT, GET_PRODUCT_WITH_TEAMS, ...` — every operation that
> spreads the `VmmBusinessPartnerOnProduct` fragment), one row for `resources.mappings.team` (Story:
> `NEW-PRODUCT-BE-G-??`, External: —, New: Yes, Impacts: the 2 operations that select it), one row for
> `unDroppablePartners` (Story: `PRODUCT-BE-G-07`, External: —, New: No, Notes: 🔬 SPIKE-04, Impacts:
> the operations selecting it), one row for the root `getProduct` field itself (Story:
> `PRODUCT-BE-B-01`, Impacts: all 12 operations), and so on for every other distinct
> field/entity/root the story's operations touch. Every genuinely plain field resolver still gets its
> own row (Story: the real id, External/New: —/No) — the table's job is completeness, not brevity by
> omission; brevity comes from field-indexing (no repeats), not from leaving fields out.
>
> **Before calling anything a gap: check `be-05-attribute-inventory.md` first**, not just
> `be-04-stories.md`'s prose — it's the more complete field→story source and prevents false gaps.
>
> **Actively check for missing ENTITY-RESOLVER stories, not just missing field stories.** This
> program's `H`-phase = cross-subgraph entity resolution. If a field touches an external platform
> (VMM, IG, CORONA, search, relationship, user-profile, etc.) with no `H`-phase-equivalent story the
> way attachment/discussion/sample/claim/construction/product have, that's a gap to flag.
>
> **Scope: queries only, 8 phase-1 domains** (product, bom, claims, measurement, impression,
> productDetails, packaging, watchlist). Skip mutations (note excluded count in one line). A fragment
> is not a separate row category — fold its fields into the table like any other field, tagged with
> which fragment they came from in the Notes column if useful.
>
> **Do NOT invent new story ids.** Reuse exactly what exists. A genuine gap gets a placeholder
> `NEW-<DOMAIN>-BE-<phase>-??` — never a fabricated real-looking id, never written into
> `be-04-stories.md`/Jira/Confluence.
>
> **Sources (check in this order):**
> 1. [`fe-08-frontend-stories.md`](../../analysis/program/fe-08-frontend-stories.md) — FE story + Operations line.
> 2. The client `.txt` file(s) under `ClientCallingGqlQueries/` — confirm what each operation actually touches (don't rely on the inventory's field-count alone).
> 3. [`fe-03-merged-inventory.md`](../../analysis/program/fe-03-merged-inventory.md) + [`fe-11-traceability-matrix.md`](../../analysis/program/fe-11-traceability-matrix.md) — operation → BE story, starting point.
> 4. `output/analysis/<domain>/be-05-attribute-inventory.md` — check before concluding a gap.
> 5. `output/analysis/<domain>/be-04-stories.md` — Depends-on chain, spike/ADR gates.
> 6. `output/analysis/<domain>/be-03-schema.graphql` — confirm `@key`/`@extends` shape for federated references.

## Prompt

Replace `<FE_STORY_ID>` with one story id, or say "all in-scope FE stories" to run once per
query-bearing phase-1 FE story (skip pure-mutation/cross-cutting stories — no page for those).

```
For <FE_STORY_ID>, produce output/clientStoryDependency/<FE_STORY_ID>.md — ONE field-indexed table,
no narrative, no separate tables:

## <FE_STORY_ID> — <story title>
Queries in scope: <op1>, <op2>, ... · Mutations excluded: <count>

**Must complete first:** BE — <every distinct real BE story id in the table, comma-separated> ·
Spikes — <every distinct SPIKE-0x, omit "Spikes —" if none> · New — <every distinct NEW-...-??
placeholder, omit "New —" if none>

| Field/Entity | Story | External Dep | New? | Impacts (queries) | Notes |
|---|---|---|---|---|---|

One row per DISTINCT field, entity, or root query field this story's operations touch — not one row
per operation. If the same field/entity is selected by multiple operations, it is still ONE row, with
every operation listed in "Impacts."

"Field/Entity" = the field/entity name (e.g. `getProduct` for the root, `VMM_BusinessPartner`,
`unDroppablePartners`, or a plain scalar group like "id/description/createdAt (~60 scalars)" when
many trivial fields share one story — group only truly interchangeable trivial scalars, never group
a field that has its own spike/external/gap status with one that doesn't).
"Story" = the real BE story id, or the `NEW-<DOMAIN>-BE-<phase>-??` placeholder for a genuine gap.
"External Dep" = the service name (VMM/IG/CORONA/APEX/materialHub/search/rating/relationship/
user-profile/discussion·attachment) if this field's resolution depends on it, else "—". If New=Yes
AND this is external, ALSO list every other existing story that depends on the same service (e.g.
"VMM — also PRODUCT-BE-G-07, G-13, F-10") so the full blast radius is visible in this one row.
"New?" = Yes (no story exists — this row's Story column is a placeholder) or No.
"Impacts (queries)" = every query operation (from the "Queries in scope" line) that selects this
field/entity, comma-separated. This is what makes the table field-indexed instead of query-indexed —
it was NOT scoped per-operation before, so make sure every row states this precisely.
"Notes" = spike gate (`🔬 SPIKE-0x`), the fragment name this field came through if relevant, or
anything else short and specific — one clause, not a paragraph. Leave blank if nothing to add.

Do not fabricate a story id or gap-fill a plausible-sounding story. "Not found in existing
inventory" beats a guess, always.
```

## After a batch

```
Consolidate across all reports in output/clientStoryDependency/ into
output/clientStoryDependency/00-NEWLY-IDENTIFIED-STORIES.md, two short tables:

| Placeholder | What it must do | Blocks (FE stories) | Domain |
|---|---|---|---|

| External service | Affects (FE stories) |
|---|---|

Dedupe both by id/service (the same gap or service recurring across stories is ONE row listing every
FE story it touches, not one row per story). Rank the first table most-blocking first.
```
