# Additional Network Call Analysis — Frontend

> Phase 7 deliverable · Hand-authored (not regenerated by the pipeline) · 2026-07-16
> Scope: the 8 phase-1 domains. Companion docs: [05-federation-impact.md](./05-federation-impact.md), [06-ui-impact.md](./06-ui-impact.md)

## 1. Baseline

- Today: 142 phase-1 frontend operations post to the single spark-internal-graphql gateway; each document is one HTTP round trip, resolved monolithically.
- Target: the same documents post to the federated router; the router plans entity fetches across subgraphs (`plm-product`, `spark-claims`, later others).
- Default position: **no additional frontend-issued requests** — fan-out moves server-side into the router's query plan.

## 2. Where the request count actually changes

### 2.1 Cross-domain documents split across migration waves

- 4 phase-1 documents mix root fields from different domains or scopes:
  - `getBomDataAndImpressions` — BOM + Impression roots.
  - `getCarryForwardFormData` — Product + Impression roots.
  - `getBulkDiscussionData` — Product + access-control roots (access control is out of scope in phase 1).
  - `getFormData` — Product + tags roots (tags out of scope in phase 1).
- While one root's domain is on the router and the other is not, the document must either:
  - stay on the legacy gateway until both sides land (preferred — zero extra calls), or
  - split into two documents (one per endpoint) — **+1 request** per render for the interim.
- Story-level decision recorded per document in [08-frontend-stories.md](./08-frontend-stories.md).

### 2.2 TechPack counts (facade-then-federate — ADR-015 Option B, 🟠 draft)

- `getProductTechPackCountV1` / `getProductTechPackBulkCountV1` aggregate across 7 sequential elastic queries backend-side today.
- Facade step: request count unchanged (1 call), latency owned by the facade.
- Federate step: counts become resolved fields on `Product` — the dedicated count queries can merge into existing product documents, **removing** up to 2 requests on product-list screens.

### 2.3 Multi-step writes return richer results

- `updateBom`, `updatePackaging`, `updateClaim`, `updateWatchlistEntries`, `updateProductDetailsSet`, `putSampleMeasurementSet`, partner drop/undrop (ADR-012/013 — 🟠 draft):
  - Today some flows refetch the parent document after mutation because the mutation response is thin.
  - Target mutation payloads should return the updated entity; each adopted case **removes** one refetch round trip.
  - Frontend stories require `update`-function cache writes instead of blanket `refetchQueries`.

### 2.4 Newly hydrated entities remove client-side lookups

- Flat ids that become entities (`createdBy`, `brand`, `department`, `clazz`) are hydrated in the same round trip by the router.
- Screens that today issue follow-up queries to resolve display names (user, brand) can drop those — **−1 request** per affected screen.

## 3. Latency and batching considerations

- Router fan-out: a `getProduct` document touching claims/bom/attachment entities becomes parallel subgraph fetches — p50 likely flat, p95 governed by the slowest subgraph; the 7 elastic queries behind TechPack are **sequential** today, so the facade keeps that cost until federate.
- Apollo batching assumptions: `spark-internal-graphql` batches at the gateway; the router has its own batching — the frontend keeps one-document-per-request semantics and needs no client batching changes.
- Partial responses: subgraph failures surface as `errors[]` + partial `data`; screens composed from multiple entities must render partial data instead of a full-page error (error-policy `all` per migrated query).

## 4. Caching strategy changes

- `__typename` renames invalidate every cached object identity at cutover — plan a cache reset (or versioned cache key) with the endpoint flag flip; do not attempt in-place migration of persisted cache state.
- `keyFields` must mirror federation `@key`s (`Product.id`, `Bom.id`, `Claims.id`, `TeamV2.teamId`, `Discussion.discussionId`).
- `possibleTypes` regenerates from the router schema (impression sub-types, `IG_Clazz_Filter` polymorphism — ADR-017 scope).
- Normalization win: entities shared across domains (Product inside claim/watchlist/bom screens) dedupe in cache once typenames unify — fewer refetches after navigation.

## 5. Lazy-loading opportunities

- Wide documents are the cheapest wins once fields become entity-resolved:
  - `getProductScaffolding` (131 fields) and `getFormData` (132 fields) can defer sibling-domain entities (`@defer` when router support lands, or split secondary queries triggered on tab focus).
  - `getProductWithAttachmentsAndComponents` (52 fields) — attachments panel is below the fold on the product page; candidate for on-demand fetch.
- Recommendation: do not bundle lazy-loading into cutover stories — record as follow-up optimizations to keep cutover diffs mechanical.

## 6. User-experience impact summary

| Change | Direction | Where |
|---|---|---|
| Interim split of cross-domain documents | +1 request (temporary) | 4 documents (§2.1) |
| TechPack counts federated onto Product | −1 to −2 requests | product list / detail |
| Mutation responses replace refetches | −1 request per write flow | 7 multi-step writes |
| Hydrated entities replace lookups | −1 request per screen | user/brand/department displays |
| Router fan-out | latency profile shift, not count | wide product documents |
