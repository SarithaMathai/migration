# Federated GraphQL Impact Assessment — Frontend

> Phase 5 deliverable · Hand-authored (not regenerated by the pipeline) · 2026-07-16
> Data source: [03-merged-inventory.md](./03-merged-inventory.md) · Target schemas: `output/initial-analysis/{domain}/03-schema.graphql`

## 1. Structural differences — current schema vs federated schema

### 1.1 Gateway and endpoint

- Today: every frontend operation is posted to the `spark-internal-graphql` Apollo gateway endpoint.
- Target: one federated router endpoint; subgraphs (`plm-product`, `spark-claims`, later `plm-workspace`, `plm-attachment`, …) are invisible to the client.
- The frontend never calls a subgraph directly — the URL swap is a single Apollo `HttpLink` change, gated by a feature flag during dual-run.

### 1.2 Type renames — `SPARK_` prefix dropped on owned types

- Every owned type loses the `SPARK_` prefix in the federated schema: `SPARK_Product` → `Product`, `SPARK_Bom` → `Bom`, `SPARK_Claims` → `Claims`, `SPARK_Measurements` → `Measurements`, `SPARK_Packaging` → `Packaging`, `SPARK_Watchlist` → `Watchlist`, `SPARK_Impression` → `Impression`, `SPARK_ProductDetail` → `ProductDetail`.
- Platform-stitched stubs keep their prefixes (`VMM_*`, `IG_*`, `CORONA_*`, `DOPPLER_*`).
- Frontend blast radius of the rename:
  - 22 client fragments declare `on SPARK_*` type conditions for phase-1 types — each fragment must be re-targeted.
  - Apollo cache `typePolicies` and `possibleTypes` are keyed by `__typename` — cache normalization breaks silently if not remapped.
  - Generated TypeScript types (`SparkProduct`, …) change names — every import site recompiles.

### 1.3 Flattened fields become entity references

- Current schema (example):

```graphql
type SPARK_Product {
  thumbnailId: String
  parentProductId: String
  createdBy: String
}
```

- Federated schema (example):

```graphql
type Product @key(fields: "id") {
  createdBy: UserProfileAttributes   # hydrated entity, user-profile subgraph
  brand: VMM_Brand                   # @key(fields: "brandId") stub
  department: IG_Department          # @key(fields: "id") stub
}
```

- Pattern for the frontend:
  - Flat scalar id fields become nested selections (`createdBy` → `createdBy { id displayName }`).
  - Components that displayed a raw id and resolved the display name via a second lookup can drop that lookup.
  - Components that only need the id must now select `entity { id }` — one extra nesting level in every consuming selection and TypeScript model.

### 1.4 Cross-domain fields become subgraph entity stubs

- On `Product`, sibling-domain data (`boms`, `claims`, `measurements`, `packagings`, `attachments`, `teams`, `discussions`, `workspaces`) resolves through `@key`-stitched entities owned by other subgraphs.
- Frontend effect:
  - Existing nested selections keep working — the router performs the entity fetches.
  - Latency profile changes: one client request may fan out to several subgraphs (see [07-network-call-analysis.md](./07-network-call-analysis.md)).
  - Error handling changes: partial data with `errors[]` per subgraph replaces all-or-nothing responses.

### 1.5 Resolver ownership changes

- Ownership after migration (from the merged inventory):
  - `plm-product` (host): product, bom, measurement, productDetails, packaging, watchlist, impression.
  - `spark-claims` (separate service): claims.
  - Later phases: workspace, attachment, discussion, sample, search subgraphs.
- The frontend does not see resolver ownership directly, but story sequencing does: a frontend cutover story can start only after the owning subgraph's backend story is delivered (dependency matrix: [09-story-dependency-matrix.md](./09-story-dependency-matrix.md)).

## 2. Migration impact catalogue

| Impact class | Where it hits | Frontend consequence |
|---|---|---|
| Type rename (`SPARK_*` →) | all 8 domains, 229 fragments | Fragment type conditions, codegen types, cache keys |
| Field nesting (flat id → entity) | product, bom, claims | Selection rewrites + TypeScript model updates |
| Removed / deprecated fields | see §3 | Replace before cutover — federated schema will not carry them |
| Newly hydrated entities | `createdBy`, `brand`, `department`, `clazz`, partners | Optional simplification: drop client-side lookups |
| Entity references (`@key`) | `Product`, `Bom`, `Claims`, shared types in [04-domain-analysis.md §Shared object types](./04-domain-analysis.md) | `possibleTypes` / `keyFields` config, fragment reuse across subgraphs |
| Aggregated counts via facade | TechPack (`getProductTechPackCountV1`, `getProductTechPackBulkCountV1`) | Keep operation shape during facade phase; re-point later (facade-then-federate, ADR-015 Option B — 🟠 draft, ratification pending) |
| Multi-step writes | `updateBom`, `updatePackaging`, `updateMeasurement`, `updateClaim`, `updateWatchlistEntries`, `updateProductDetailsSet`, partner drop/undrop | Mutation response shapes may gain saga status fields (ADR-013 — 🟠 draft); optimistic-update logic must be revisited |
| Polymorphic resolution | Impression sub-types (5), `IG_Clazz_Filter` | `possibleTypes` map must be regenerated from the federated schema |

## 3. Deprecated fields still selected by the frontend (phase-1 domains)

- The federated schema does not carry deprecated fields — each usage below must be replaced before that domain's cutover.

| Field | Deprecation note | Domain | FE replacement action |
|---|---|---|---|
| `SPARK_Measurements.humanId` | Use `parentId` | measurement | Swap selection + rendering to `parentId` |
| `SPARK_Dieline.attachments` | deprecated | packaging | Move to attachment entity selections |
| `SPARK_PackagingFieldValues.resourceType` | deprecated | packaging | Drop from selections; derive from context |
| `SPARK_ProductRules.businessPartners` | use rule attribute | product | Re-read from the rule attribute shape |

- Later-phase domains also consume deprecated fields (trim facility row-ids, material-hub supplier fields, `SPARK_SpgFileLibrary.*`, `accreditation`) — recorded in [02-backend-schema-inventory.md §Deprecations](./02-backend-schema-inventory.md) for their phase.

## 4. Per-domain impact (phase 1)

### Product — 🔴 High

- 64 frontend operations, ~50 consuming components/hooks across 8 client libraries.
- `getProduct` alone backs 20+ distinct client documents (breadcrumbs, discussions, packaging DPCI, files, teams, scaffolding) — the single highest-risk cutover surface.
- TechPack count queries follow facade-then-federate: frontend keeps current shapes in the facade step, then migrates once (ADR-015 Option B — 🟠 draft).
- Partner drop/undrop mutations (`productBusinessPartnerActions`, `updateBusinessPartnerStatuses`) are orchestrated multi-step writes (ADR-012 — 🟠 draft); mutation payloads and error surfaces may change.
- Cross-domain documents (`getBulkDiscussionData`, `getFormData`, `getCarryForwardFormData`) span product + access-control/tags/impression root fields — they split across waves and need interim dual-endpoint handling.

### BOM — 🔴 High

- 19 operations, 12 components/hooks.
- `updateBom` is a 3-step write with no rollback today (ADR-013 — 🟠 draft) — frontend must handle partial-failure responses explicitly.
- `BUILD_FILES_FRAGMENT` / `BOM_FABRIC_SPEC_COMBO_DETAILS` are runtime-composed fragment factories — incompatible with codegen and persisted queries; must be statically expanded before migration.
- `getBomDataAndImpressions` couples BOM and Impression root fields in one document.

### Packaging — 🟡 Medium

- 21 operations, 15 components/hooks; heaviest mutation surface after product (9 mutations).
- `updatePackaging` multi-step write + elastic-search read cutover (`getPackagings` variants).
- Two deprecated packaging fields still selected (§3).

### Claims — 🟡 Medium

- 14 operations; separate `spark-claims` subgraph — first cross-subgraph entity test for `Product ↔ Claims`.
- `FULL_CLAIM_DETAILS(includeWorkspaces)` is a parameterized fragment factory (conditional `workspaces` selection) — must become two static fragments.
- `updateClaim` proxy-ACL multi-step write (ADR-013 scope) + camelCase response bug documented backend-side; FE must not codify the buggy shape.

### Measurement — 🟡 Medium

- 12 operations; deprecated `humanId` selection to replace (§3).
- Template fragments (`sizeTemplateFragmentWithRows`, `measurementTemplateFragment`) are defined outside the snapshot libraries — inventory gap to close during implementation.

### Product Details — 🟡 Medium

- 7 operations; `updateProductDetailsSet` multi-step write; low read surface.

### Watchlist — 🟡 Medium

- 5 operations in a single client library — cleanest cutover candidate; `updateWatchlistEntries` multi-step write is the only complexity.

### Impression — 🟢 Low

- 2 read documents, both cross-domain (`getBomDataAndImpressions` with BOM, `getCarryForwardFormData` with Product).
- Impression sub-type polymorphism (5 types) requires a regenerated `possibleTypes` map (ADR-017 — 🟠 draft).

## 5. Scope boundary

- Program scope: only the 8 phase-1 domains' queries, mutations and fields are assessed here.
- 618 further client operations resolve to later-phase domains or to services outside spark-internal-graphql (combination, materials, samples, discussions v2/V3, insights, wash-federated) — out of scope; they migrate with their own subgraph phase.
- Boundary risk to manage inside phase 1: cross-domain documents (`getBulkDiscussionData`, `getFormData`, `getCarryForwardFormData`, `getBomDataAndImpressions`) mix in-scope and out-of-scope root fields — each needs an explicit split or dual-routing decision in its story.
