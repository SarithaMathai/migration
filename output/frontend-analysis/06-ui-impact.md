## 1. Impact categories

Every migrated operation is assessed against this checklist; the per-domain sections below state which categories apply.

| Category | What changes |
|---|---|
| Component changes | Prop shapes, rendering of nested entities instead of flat ids |
| Hook updates | `useQuery`/`useMutation` generics, variable shapes, result mapping |
| Apollo cache | `typePolicies` keyed by new `__typename`s, `keyFields` for `@key` entities, `possibleTypes` regeneration |
| Fragment updates | Type-condition renames (`on SPARK_Product` → `on Product`), static expansion of runtime fragment factories |
| TypeScript models | Regenerated types; every import site of a renamed type recompiles |
| Codegen | Introspect the federated router schema instead of spark-internal-graphql |
| Query / mutation updates | Selection rewrites for nested entities, removed deprecated fields |
| State management | Optimistic updates and refetch queries tied to mutation response shapes |
| Forms & validation | Input type renames (`SPARK_*Input` →), multi-step-write status handling |
| Error handling | Partial data + per-subgraph `errors[]` instead of all-or-nothing |
| Unit / integration / snapshot tests | MockedProvider mocks re-shaped; snapshots with `__typename` churn |

## 2. Cross-cutting impacts (apply to every domain)

- One-time platform work, done once and reused by all domain cutovers:
  - Router endpoint switch behind a feature flag (dual-run capable).
  - Codegen pipeline re-pointed at the federated schema; `possibleTypes` map regenerated.
  - Apollo cache `typePolicies` remapped for renamed `__typename`s; cache eviction strategy for the cutover window.
  - Shared fragment library rename sweep (229 fragments).
- Test estate: every `MockedProvider` mock embeds operation shapes — mocks migrate with their operation, snapshots regenerate in the same PR.

### Product — 🔴 High

- Applies: all categories.
- Drivers:
  - `getProduct` variants back breadcrumbs, discussion panes, file trees, team tabs, packaging DPCI lookups — regression surface spans 5 client libraries beyond `product-queries`.
  - Partner drop/undrop mutations change optimistic-update and refetch patterns (multi-step write status).
  - TechPack count components keep their query shape during the facade step — one later re-point story instead of two rewrites.
- Watch item: `getProductScaffolding` selects 131 fields and `getFormData` 132 — widest selections in the estate; any nesting change fans out through their TypeScript models.

### BOM — 🔴 High

- Applies: fragments (heavy), components, hooks, cache, forms, error handling, tests.
- Drivers:
  - Runtime fragment factories must be statically expanded first — they block codegen for every document that embeds them.
  - `updateBom` partial-failure handling is new UI behaviour (saga status), not just a shape change.

### Packaging — 🟡 Medium

- Applies: components, hooks, mutations, forms, cache, tests.
- Drivers: 9 mutations including bulk paths; dieline evaluation flow; 2 deprecated field selections to replace.

### Claims — 🟡 Medium

- Applies: fragments, hooks, cache (cross-subgraph `Product ↔ Claims` entity), error handling.
- Drivers: `FULL_CLAIM_DETAILS(includeWorkspaces)` factory → two static fragments; template-library clone flows in `spark-legacy` consume claim fragments too.

### Measurement — 🟡 Medium

- Applies: queries, fragments, TypeScript models, tests.
- Drivers: `humanId` → `parentId` swap in selections and any UI that renders the id; template fragments defined outside the inventoried libraries need locating.

### Product Details — 🟡 Medium

- Applies: mutations, forms, error handling.
- Drivers: `updateProductDetailsSet` multi-step write; small read surface (2 queries).

### Watchlist — 🟢 Low

- Applies: queries, mutations, TypeScript models.
- Drivers: single client library, 4 consuming components; `updateWatchlistEntries` saga handling is the only nontrivial change.

### Impression — 🟢 Low

- Applies: cache (`possibleTypes` for 5 sub-types), query updates.
- Drivers: both documents are cross-domain — they migrate when their partner domain migrates, not independently.

## 4. Impact-level definitions

- **None** — operation unchanged in the federated schema; only the endpoint moves.
- **Low** — mechanical rename/regenerate; no behavioural change; single PR per operation cluster.
- **Medium** — selection or response-shape rewrites; consumer components need review; mocks and snapshots regenerate.
- **High** — behavioural change (saga status handling, cross-subgraph partial errors, dynamic-document refactor) or a regression surface spanning multiple client libraries.
