# Complex Stories — problem briefs + draft ADRs + story breakdowns

One folder per genuinely complex, cross-domain case found during the resolver analysis.
Each folder holds:

- **`00-overview.md`** — the problem brief: **§1 the problem** (problem statement · current state &
  root cause · impact · objectives, grounded in the legacy code) · **§2 what must be decided** (the open
  questions the owning spike answers, plus a pointer to the draft decision).
- **`01-adr-<case>.md`** — the **draft ADR**: today's behavior (evidence), decision drivers, options with
  trade-offs, the proposed decision with pin-downs, consequences, and the on-acceptance lifecycle. Each ADR
  assumes ADR-019's Mid-Request ACL Update mechanism wherever ACL is actually in play for that case (see
  each ADR's own ACL note) — there is no separate no-ACL scenario file; that hypothetical was retired once
  ADR-019 ratified the actual mechanism.
- **`01-stories.md`** — the **story breakdown**: which concrete story ids (in each affected domain's
  `be-04-stories.md`) implement the decision, cross-referenced by pin-down. Full story text (Current
  Behaviour, Target, Acceptance Criteria, Test Cases) lives in the domain file, not duplicated here.
- **`<case>.csv`** — generated by [`generate.py`](./generate.py) from `01-stories.md`; imported into Jira
  **separately** from the domain CSVs, as sub-tasks nested under the case's home-stub story (never
  double-counted against the domain/program story totals).

All draft ADRs are **🔴 Proposed — pending ratification**. On acceptance they move to `adrs/` and the
affected domain stories are updated, per `fedMigrationScripts/reference/SPIKE-ADR-LIFECYCLE.md`.
